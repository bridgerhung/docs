本次更新的核心方向

一、讓學生在使用系統前，清楚知道系統會蒐集哪些資料、如何使用與保存，以及使用 AI 功能時需要注意的事項。

二、把「法定個資告知、系統使用者條款確認、更新初始密碼」整合成一套不可略過的首次登入流程。

三、補強登入、作業開放時間、錄音處理、AI 使用量與名單匯入等後端防護，降低異常操作造成的風險。

四、集中法遵、題目、音訊、API 額度與帳戶設定的治理責任，移除散落於各處的重複判斷。

一、為什麼要做這次更新

這套系統會保存學生的帳號資料、英文作答、錄音、AI 回饋、學習歷程、心得與教師評分，也會使用外部雲端與 OpenAI API 完成部分功能。因此，正式讓學生使用前，不能只有「帳號可以登入、功能可以操作」，還必須把資料用途、保存方式、使用規則與學生權利交代清楚。

這一輪更新沒有改變 7 月 19 日已完成的教學主流程，而是補上正式使用所需的告知、確認與安全邊界。

二、為什麼系統使用者條款不能省略

更精確地說，這次建立的不是籠統的兩項說明，而是三項目的不同、不能互相取代的正式法遵機制，並由實際的資訊安全與隱私控制支撐：

法定告知

個人資料蒐集、處理及利用告知事項

說明由誰蒐集、蒐集目的與資料類別、利用期間／地區／對象／方式、學生權利，以及不提供資料的影響。

可接受使用

系統使用者條款

本系統正式名稱為「AI 英語面試練習系統使用者條款暨可接受使用規範」，界定帳號安全、授權用途、禁止行為、AI 限制與違規處理。

研究倫理

研究參與者知情同意書

決定是否參與研究、哪些資料可納入研究、如何退出及資料如何保存；必須依計畫與倫理審查要求另外辦理。

技術與管理措施

資訊安全與隱私控制

包括權限、工作階段、操作紀錄、資料保存與刪除、事故處理、風險評估及持續改善；勾選條款不能取代這些實際控制。

中華民國法律與教育部計畫的直接依據

  • 《個人資料保護法》第 8 條要求直接向當事人蒐集個人資料時，必須明確告知蒐集者名稱、目的、資料類別、利用期間／地區／對象／方式、當事人權利及不提供資料的影響。這是「個人資料告知事項」不能省略的直接法律依據。
  • 《個人資料保護法施行細則》第 12、16 條進一步說明，安全維護應包含技術與組織措施、風險管理、事故應變、內部程序、教育訓練、稽核、軌跡與證據保存；法定告知也可以電子方式完成。
  • 《教育部補助大專校院教學實踐研究計畫作業要點》第 14 點要求：涉及人體研究者，依審查結果於執行前檢附倫理審查核准；涉及以個人／群體為對象、互動介入或可識別資料之研究者，應於執行前檢附招募方式與告知同意內容。
  • 若個案屬於《人體研究法》規範的研究且不屬免審／免同意範圍，第 5、12、14 條要求研究開始前完成倫理審查，並依核准方式取得知情同意及告知研究目的、風險、權益、資料保護、退出方式與保存規劃。
重要結論：為保障使用者權益並落實個人資料保護、研究倫理、帳號安全及系統治理要求，本系統制定「系統使用者條款暨可接受使用規範」，明確說明帳號使用、錄音與作答內容、AI 功能限制、禁止行為、違規處理及聯絡管道。系統使用者條款、個資告知事項與研究參與者知情同意各有不同功能，應分別提供、分別確認並保留版本紀錄。

三、學生登入與帳戶設定

首次使用前，先完成法定個資告知與使用者條款確認

尚未完成確認的學生登入後，系統會顯示「個人資料蒐集、處理及利用告知事項」與「AI 英語面試練習系統使用者條款暨可接受使用規範」。學生必須分別勾選「已閱讀並了解個資告知」及「已閱讀、了解並同意遵守使用者條款」，系統才會接受帳戶設定。

系統會分別記錄個資告知版本與閱讀確認時間，以及系統使用者條款版本與同意時間；日後有重大修訂時，既有使用者會只需確認新版本，不必為此再次變更密碼。

法遵文件的標題、網址與版本由後端統一提供。若學生送出確認時文件版本已更新，系統會拒絕舊版本並重新載入現行文件，不會把舊畫面的勾選記成新版本確認。

閱讀：個人資料蒐集、處理及利用告知事項 閱讀：AI 英語面試練習系統使用者條款暨可接受使用規範

初始密碼與管理員重設密碼後，必須先更新

新建立的帳號會被標記為需要更新初始密碼；管理員日後協助重設密碼時，也會重新啟用這項要求。學生完成密碼更新前，後端會直接阻擋其他學生功能，不只是在畫面上顯示提醒。

管理員重設密碼時，原本已登入的裝置會一併失效，避免舊工作階段在密碼變更後仍能繼續存取帳戶。

帳戶設定視窗重新設計

帳戶設定視窗現在完整處理三種狀態：只需更新密碼、只需確認新版法遵文件，以及兩者都需要。管理員重設密碼後不會重複要求仍有效的法遵確認；只因法遵改版而需要重新確認時，也不會要求再次變更密碼。

新版也改善手機版的捲動、固定操作按鈕、密碼顯示／隱藏與欄位說明，讓學生在較小的螢幕上仍能完整完成設定。

需要完成密碼或法遵設定時，視窗不可略過，作業列表也不會先行載入；完成後才恢復正常作業流程，避免短暫錯誤、重複提示或錯誤跳轉。

系統使用者條款確認不等於同意參與研究

這次線上確認的目的，是完成系統可接受使用規則與個人資料處理所需的告知。是否參與教育部教學實踐研究，以及哪些資料可以作為研究資料，仍以另行辦理並經適用之研究倫理程序確認的研究參與者知情同意內容為準；兩者不會混在一起。

四、登入、資料與 API 安全加固

學生工作階段改由伺服器完整驗證

學生與管理員的登入憑證現在都必須對應到仍有效的伺服器工作階段。學生於另一個裝置登入、密碼被重設或工作階段被撤銷後，舊憑證就不能只靠尚未到期的 cookie 繼續使用。

同時發生的登入要求也會依序處理，降低兩個登入流程互相覆蓋、產生不一致狀態的可能性。

尚未開放的作業，統一由後端把關

作業的開始時間現在套用在更多讀取與寫入入口。即使有人略過前端畫面、直接呼叫網址或 API，也不能提前建立職位資料、寫入練習內容或讀取尚未開放的作業。

錄音先檢查，再進入語音與 AI 處理

系統會在錄音送進 ffmpeg、語音辨識或遠端處理前先檢查實際時長；若使用遠端音訊 worker，也會在另一端再次確認。格式異常或超過限制的檔案會提早拒絕，避免不必要的資源消耗。

補齊跟讀階段的 API 額度控管

跟讀階段的語音產生與辨識現在也納入每小時及每日使用量限制。額度檢查與紀錄會在同一個受控流程中完成，降低多個請求同時進入時超出設定上限的情況。

管理端 XLSX 匯入範圍更明確

學生名單匯入限定使用 A 至 E 欄，單次最多 200 筆學生資料；超出範圍或宣告異常巨大工作表的檔案會提早停止處理。這項調整可避免格式錯誤或特殊試算表長時間占用伺服器資源。

五、題目、語音與流程治理

自訂面試題維持英文任務與教師順序

管理員自訂題組必須恰好四題，每題 1 至 500 個字元，並使用英文或其他拉丁文字；教師輸入的順序會直接保留。學生的職位名稱、工作說明與能力背景仍可使用中文，不受面試題語系限制。

角色扮演若產生過長、非拉丁文字或偏離指定題意的問題，系統會回到原本且已驗證的核心題目，不再以失去原題意的通用問句取代。

文字轉語音由用途 profile 統一治理

角色扮演與跟讀分別使用固定的語音 profile；主程式與遠端音訊 worker 接受相同的 text 與 profile 契約。單次文字轉語音輸入上限統一為 4,096 個字元，避免超過供應商介面限制。

模組責任集中，不再散落重複補丁

法遵版本與寫入、題目規則、音訊物件命名、API 階段與額度鎖，以及帳戶設定畫面，現在都由各自的服務或 controller 管理。測試環境的資料庫 migration 也整理為單一現況基線，不保留不再需要的相容分支。

六、公開版原始碼與部署邊界整理

為日後提供乾淨的公開版原始碼，這一輪也持續整理公開設定與內部資料的邊界，包括移除或排除不應隨公開套件提供的部署設定、內部安全稽核材料、備份目錄與發版信件，並將範例網址、映像名稱及聯絡設定改為可由部署者自行設定。

系統啟動時也會檢查主要簽章密鑰與遠端 worker 密鑰，不接受過短或明顯仍是範例文字的設定；主程式與 worker-only 模式則必須使用各自正確的啟動入口。

這些整理代表公開發布邊界已更加清楚，但不等於本公告所述版本已經公開上架；正式原始碼發布仍會另外建立乾淨封裝並完成內容檢查。

  • 首次登入且仍使用初始密碼者，請在閱讀法遵事項後設定新的登入密碼；僅因法遵事項改版而需重新確認的既有使用者，不必再次變更密碼。
  • 如果密碼由管理員重設，舊裝置會登出；請使用新取得的密碼登入並完成更新。
  • 這次更新不會刪除既有作答、錄音、AI 回饋、心得或教師評分。

九、Clair 學姐的測試回饋與處理說明

本次特別邀請陽明交大技職所博士二年級的 Clair 學姐協助測試。她實際操作後認為，系統整體流暢、介面直觀，也能有效協助學生練習；名琮再從回饋中整理出兩個仍需要改進的重點，處理情況如下：

Clair｜語調調整｜已完成

Clair 認為，跟讀與角色扮演可以協助初階學生，也能提升口說練習意願；但目前語音的語調仍偏呆板，和真實面試中自然的說話方式有些距離。

目前已依現有 TTS 架構，為角色扮演與跟讀分別加入語音提示。角色扮演語音會著重自然的語調、停頓、重音與合宜的情緒；跟讀示範則兼顧清楚度與自然節奏，希望在不改動原文的前提下，減少機械感。

Clair｜協助學生建立對職務的想像｜替代方案

Clair 指出，學生最需要額外協助的可能是「填寫目標職位資料」階段。這個問題不只是欄位提示要怎麼寫，而是學生對自己了解多少、是否曾探索想應徵的職位，以及能不能把對未來的想像整理成具體的工作內容與能力需求。

如果要讓系統主動搜尋最新職缺、判讀外部資料，再把資訊帶回本站進行個別化引導，已經涉及職涯輔導。以目前可投入的人力、資源與後續維護能力，尚不足以把這項能力穩定且負責任地整合進系統，因此本期傾向不直接實作。

現階段會改採課堂前導作為替代方案。我會請計畫主持人特別關注這個環節，在學生進入系統前，先於課堂協助他們思考自己的未來、探索可能的職涯方向，並對目標職務建立基本想像，再把整理後的資訊帶入系統練習。

這一輪更新做的多半不是學生練習時最顯眼的功能，卻是系統要真正進入課程前一定要完成的基礎工作。接下來仍會持續檢查正式環境、操作說明與部署流程，讓學生和老師在開始使用時，有清楚的規則，也有比較完整的安全保障。
